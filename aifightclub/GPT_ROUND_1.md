# GPT Round 1 — Phase 1 First Audit

> **Author:** Codex (GPT 5.4 Extra High)
> **Date:** 2026-03-25
> **Baseline:** `npx tsc --noEmit` passes. `pnpm test` passes with 39 test files and 799 total tests (795 passed, 4 skipped). `package.json` is `0.16.0`.
> **Scope:** Phase 1 deep code audit of the DSP path, with one Phase 2 finding parked when it surfaced directly from the code and tests.

## [PHASE 1] Finding: Worker restart replays stale settings after a crash
**Author:** Codex | **Status:** Open
**Evidence:** `hooks/useDSPWorker.ts:195-226` only replays `lastInitRef.current` on auto-restart; `hooks/useDSPWorker.ts:257-272` updates `lastInitRef` only inside `init()`; `hooks/useDSPWorker.ts:277-280` forwards `updateSettings()` without updating that restart snapshot; `hooks/useDSPWorker.ts:249-253` drops non-`init`/`reset` messages while `isReadyRef` is false; `hooks/useAudioAnalyzer.ts:309-316` pushes the full live settings object on every settings change only to the currently running worker.
**Finding:** The restart path has no authoritative “latest settings” source. After the initial `init()`, subsequent settings changes are sent to the live worker but never copied into `lastInitRef`. If the worker crashes later, the restarted worker boots with the stale settings captured at the last `init()` call, and any `updateSettings` emitted during the not-ready window are silently dropped.
**Impact:** Silent state divergence after crash recovery. The UI can continue to show the latest settings while the worker classifies with older thresholds, enabled algorithms, merge tolerances, and room settings until the user changes settings again. This is the worst kind of failure for a live-sound tool because it looks healthy while making decisions with the wrong config.
**Proposal:** Keep a separate authoritative `latestSettingsRef` inside `useDSPWorker`, update it on every `init()` and `updateSettings()`, and use that ref for restart re-initialization. Also queue or coalesce `updateSettings` messages while the worker is not ready so post-crash settings changes are replayed once the worker posts `ready`.
**Devil's Advocate:** Worker crashes may be rare, and if they happen immediately after start before any settings changes, the restart snapshot is accurate. If the team treats crashes as terminal failures instead of “recover and continue,” the practical exposure is smaller. The current code, however, clearly advertises auto-recovery, so the recovered state needs to be trustworthy.

## [PHASE 1] Finding: Auto-restarted workers are not terminated on unmount
**Author:** Codex | **Status:** Open
**Evidence:** `hooks/useDSPWorker.ts:216-221` allocates a replacement worker after a crash and stores it in `workerRef.current`; `hooks/useDSPWorker.ts:232-246` captures the original `worker` in the mount effect and terminates that closed-over instance during cleanup instead of terminating `workerRef.current`.
**Finding:** The hook’s unmount cleanup is bound to the first worker instance only. After an auto-restart, unmounting the hook terminates the dead original worker again, not the replacement worker that is actually alive.
**Impact:** Orphaned worker instances after crash + navigation/hot reload/unmount. The leak is bounded per crash, but this app is explicitly built for long-running live sessions, so “rare leak” logic is not a strong defense. An orphaned worker keeps its own `TrackManager`, `AdvisoryManager`, caches, and message handlers alive until the page goes away.
**Proposal:** In the cleanup path, terminate `workerRef.current` instead of the closed-over `worker`, and null the ref afterward. The same principle should be used anywhere the hook assumes there is only one lifetime worker instance.
**Devil's Advocate:** If the app almost never unmounts the hook after a crash, the leak may not show up often in production. But development hot reloads, route changes, and error-boundary remounts are exactly the scenarios where crash recovery tends to be exercised hardest.

## [PHASE 1] Finding: `spectrumUpdate` buffers are returned to the wrong pool
**Author:** Codex | **Status:** Open
**Evidence:** `hooks/useDSPWorker.ts:330-346` allocates `sendSpectrumUpdate()` buffers from `specUpdatePoolRef`; `lib/dsp/dspWorker.ts:352-375` returns those transferred buffers through the generic `returnBuffers` message; `hooks/useDSPWorker.ts:155-159` pushes every returned spectrum buffer into `specPoolRef`, never into `specUpdatePoolRef`.
**Finding:** The main thread has two separate spectrum buffer pools, but the return path only replenishes one of them. `processPeak()` can reuse buffers; `sendSpectrumUpdate()` cannot, because its pool is drained and never refilled.
**Impact:** Ongoing avoidable allocation churn for the periodic content-type feed. At the default cadence (`createAudioAnalyzer.ts:53` sets `spectrumUpdateIntervalMs = 500`), that is a fresh ~16 KB `Float32Array` every 500 ms at 8192 FFT size after the first three pooled buffers are consumed. Over a two-hour show, that is roughly 14,400 extra typed-array allocations and about 225 MB of cumulative spectrum-buffer churn for a path that was explicitly designed to be pooled.
**Proposal:** Distinguish `returnBuffers` by source (`processPeak` vs `spectrumUpdate`) or split them into separate outbound message types so each buffer returns to the correct pool. If the generic message is kept, include a `source` tag and route it accordingly in `useDSPWorker`.
**Devil's Advocate:** The feed only runs at ~2 fps, so this is not a catastrophic per-frame leak. The arrays are also short-lived and garbage collectors handle this class of object reasonably well. The counterpoint is that the code already paid the complexity cost of buffer pooling; misrouting the buffers defeats that design for no gain.

## [PHASE 1] Finding: Threshold-sync comments already disagree with shipped values
**Author:** Codex | **Status:** Open
**Evidence:** `lib/dsp/constants.ts:352` sets `TRACK_SETTINGS.ASSOCIATION_TOLERANCE_CENTS = 100`; `lib/dsp/constants.ts:360` sets `HARMONIC_SETTINGS.TOLERANCE_CENTS = 200` but says it is “synced with ASSOCIATION_TOLERANCE_CENTS”; `lib/dsp/constants.ts:649` sets `peakMergeCents: 100`; `lib/dsp/constants.ts:656` sets `harmonicToleranceCents: 200` and again says it is “synced with ASSOCIATION_TOLERANCE_CENTS”; `lib/dsp/trackManager.ts:33-36` actually consumes the 100-cent association value at runtime.
**Finding:** The comments claim a shared tuning relationship that no longer exists in the code. Track association and advisory merging are on a semitone, while harmonic matching is on a whole tone. That may be intentional, but the current comments say the opposite.
**Impact:** This is a maintenance booby trap. Future threshold tuning, audits, and test design will assume these knobs move together when they do not. For an app with multiple cents-based gates across detector, track manager, advisory manager, and history grouping, false “synced” comments are how subtle desynchronization survives code review.
**Proposal:** Either derive the related tolerances from a small shared constant set with explicit names, or update the comments to state the real relationship: association/merge at 100 cents, harmonic matching at 200 cents, and why they intentionally differ.
**Devil's Advocate:** Divergence may be deliberate and acoustically justified. If so, the runtime behavior is correct and the bug is documentation drift, not DSP math. That still matters here because this repository leans heavily on comment-driven tuning rationale.

## [PHASE 1] Finding: Several “critical vulnerability” tests are documentation, not regression protection
**Author:** Codex | **Status:** Open
**Evidence:** `tests/dsp/algorithmFusion.gpt.test.ts:76-85` logs a “FALSE NEGATIVE” moving-mic scenario but makes no assertion at all; `tests/dsp/algorithmFusion.chatgpt-context.test.ts:360-363` contains a pure placeholder test with `expect(true).toBe(true)`; current `pnpm test` output still prints scenario logs like `[GPT MUSIC FP] flanger: probability=0.718, verdict=FEEDBACK` and `[GPT COMPRESSED FP] AutoTune: probability=0.772, verdict=FEEDBACK`, but the suites remain green.
**Finding:** The repo has accumulated scenario tests that read like hard vulnerability guards while behaving like narrative documentation. Some only log values, some use very loose inequalities, and some are literal no-op placeholders.
**Impact:** False confidence in CI. A passing suite currently means “the code still runs,” not “the documented vulnerability envelope is preserved.” That matters because these tests are being used as evidence in tuning discussions and audit narratives.
**Proposal:** Split the intent cleanly. Keep explanatory writeups in docs, but convert genuine regression cases into exact or tightly-bounded assertions on probability, confidence, and verdict. Pure documentation tests should be deleted or moved out of the green path.
**Devil's Advocate:** DSP tuning is probabilistic, and overly rigid assertions can make the suite brittle during legitimate retuning. The compromise is to assert the actual safety property being protected instead of exact prose-level narratives or placeholder truths.

## [PHASE 2] Finding: Comb bonus can flip verdicts by construction, not just by stronger evidence
**Author:** Codex | **Status:** Open
**Evidence:** `lib/dsp/algorithmFusion.ts:779-806` doubles comb weight in the numerator (`weights.comb * 2`) but adds only the base comb weight to `totalWeight`; `lib/dsp/algorithmFusion.ts:860-902` then thresholds the resulting probability and confidence into verdicts; `tests/dsp/algorithmFusion.test.ts:400-413` encodes the exact flip case, and the current `pnpm test` run printed `[V7 COMB] noComb=0.600, withComb=0.678, delta=0.078`.
**Finding:** Comb evidence has a privileged math path: it is amplified more aggressively than the denominator acknowledges. That means comb can take an otherwise threshold-edge case and push it over the line even when the other algorithms did not become more certain.
**Impact:** Borderline verdict inflation in exactly the cases where comb is most likely to be confused by acoustically similar modulation artifacts. This does not prove the current policy is wrong, but it does prove the verdict shift is a baked-in consequence of the weighting scheme, not an emergent property of “more corroboration.”
**Proposal:** Revisit whether comb should be a post-fusion bonus/gate instead of asymmetric weight inflation, or at minimum cap how much comb alone can lift a borderline probability. If the asymmetry is intentional, document it as a policy choice rather than a neutral weighted average.
**Devil's Advocate:** A real acoustic loop does produce valuable comb evidence, and the whole point of the bonus is to let that evidence speak loudly. If Don confirms that a strong comb signature should aggressively promote alerts in real-world ring-out and monitor workflows, the current behavior may be the correct bias.
