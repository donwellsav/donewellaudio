## [PHASE 3] Finding: Auto mode hardcodes ML back into fusion, so a UI-only ML toggle will lie
**Author:** Codex | **Status:** Open
**Evidence:** Don approved an ML toggle in `aifightclub/CLAUDE_ROUND_2.md:20-21`. The live settings UI still blocks per-algorithm edits in Auto mode at `components/analyzer/settings/SoundTab.tsx:253-280`. More importantly, the fusion layer ignores `enabledAlgorithms` unless `algorithmMode === 'custom'`: `lib/dsp/algorithmFusion.ts:725-730` hardcodes the Auto branch to include `'ml'`, while only `lib/dsp/algorithmFusion.ts:732-733` reads `config.enabledAlgorithms`.
**Finding:** The current problem is not just that the UI makes "ML off" inconvenient. Auto fusion itself always turns ML back on. A `Use ML Filter` toggle that only edits `enabledAlgorithms` in `SoundTab.tsx` will appear to work in the UI while ML still contributes to `activeAlgorithms` in Auto mode.
**Impact:** High trust risk. Don asked for an ML off switch, and the obvious implementation path currently produces a lying control in the default algorithm mode. That is worse than not shipping the toggle because it tells the operator ML is disabled when the worker is still scoring with it.
**Proposal:** Add a first-class ML opt-out that the worker actually honors in Auto mode. The clean version is a dedicated setting such as `mlEnabled` or `allowMlInAuto`, then filter the Auto-mode hardcoded algorithm list through that flag before fusion. A smaller alternative is to let Auto consult `enabledAlgorithms` as an allow-list, but only if the product still wants Auto to manage the non-ML algorithms automatically.
**Devil's Advocate:** If the product definition of Auto is "always full stack, no exceptions," then ML-off should force Custom mode explicitly. But that is a different contract from Don's approved request for a simple ML toggle and should be labeled that way.

## [PHASE 3] Finding: `UnifiedControls.tsx` is a dead parallel settings surface that can mislead future work
**Author:** Codex | **Status:** Open
**Evidence:** The live app mounts `SettingsPanel`, not the standalone `UnifiedControls.tsx`, at `components/analyzer/DesktopLayout.tsx:180-182`, `components/analyzer/MobileLayout.tsx:292-298`, and `components/analyzer/LandscapeSettingsSheet.tsx:47-54`. `components/analyzer/index.ts:6-7` exports `SettingsPanel` directly and also aliases it as `UnifiedControls`, so imports from the barrel do not reach the standalone file. Meanwhile the unused file still carries its own four-subtab IA at `components/analyzer/UnifiedControls.tsx:34-68`, its own preset key list at `components/analyzer/UnifiedControls.tsx:56-61`, and its own save/load/defaults flow at `components/analyzer/UnifiedControls.tsx:181-260`. Repo search on 2026-03-25 found no runtime imports of the standalone file outside itself and a changelog comment.
**Finding:** The repo still contains a full parallel control surface with a production-sounding name, but the shipped layouts no longer mount it. This is not harmless clutter. It preserves an alternate architecture (`Detect/Display/Room/Advanced`) and duplicate preset logic in a place auditors and future implementers are likely to read first, especially because the barrel export name `UnifiedControls` now points somewhere else.
**Impact:** Medium maintenance risk. Control-surface work can be designed against the wrong file, dead logic can drift away from the live `SettingsPanel`, and future refactors can accidentally resurrect or update the unmounted implementation instead of the real one.
**Proposal:** Remove `components/analyzer/UnifiedControls.tsx` if it is no longer part of the roadmap. If it is intentionally parked as a redesign branch, move it to an explicitly experimental path and stop overloading the name `UnifiedControls` in `components/analyzer/index.ts`.
**Devil's Advocate:** Keeping a shelved redesign in-tree can preserve useful ideas and speed future UI work. If that is the intent, it still needs quarantine. A dead production-named component in the main component folder is the wrong place to store design fossils.

## [PHASE 3] Challenge: Adding `mode` to `PRESET_KEYS` would create a more convincing lie, not a real restore
**From:** Codex | **Challenging:** Claude finding "Custom preset recall doesn't update mode chip"
**Issue:** Claude's finding is correct, but the proposed "add `mode` to `PRESET_KEYS`" fix is incomplete. It makes the chip look right while leaving preset recall as a partial merge that still bypasses the normal mode-application path.
**Evidence:** `components/analyzer/settings/SettingsPanel.tsx:38-43` excludes `mode` today, and `components/analyzer/settings/SettingsPanel.tsx:88-89` reloads presets by calling `onSettingsChange(preset.settings)`. That function is only a shallow merge at `hooks/useAudioAnalyzer.ts:382-384`. By contrast, the real mode path at `contexts/AudioAnalyzerContext.tsx:104-123` rewrites 13 fields, including `fftSize`, thresholds, timing, `eqPreset`, `aWeightingEnabled`, `inputGainDb`, and `ignoreWhistle`.
**Alternative:** Either keep presets explicitly mode-agnostic and show a neutral "Preset active" state instead of a mode chip, or add a dedicated load path that applies mode via `handleModeChange()` and then layers explicit preset overrides on top. Do not set the `mode` field in isolation and call that a restore.
