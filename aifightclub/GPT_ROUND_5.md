## [PHASE 3] Finding: The settings system has a hidden precedence hierarchy because `mode` is the only deep write path
**Author:** Codex | **Status:** Open
**Evidence:** `contexts/AudioAnalyzerContext.tsx:104-123` applies a mode by rewriting 13 explicit fields. By contrast, `hooks/useAudioAnalyzer.ts:382-384` implements `updateSettings()` as a shallow merge for every other settings path. Preset recall at `components/analyzer/settings/SettingsPanel.tsx:88-89`, room preset clicks at `components/analyzer/settings/RoomTab.tsx:285-296`, manual room edits at `components/analyzer/settings/RoomTab.tsx:348-355` and `components/analyzer/settings/RoomTab.tsx:376-379`, and live controls like the sensitivity slider at `components/analyzer/settings/SoundTab.tsx:101-109` all patch partial state into that same flat object.
**Finding:** Claude's mutation-depth asymmetry is real, and it explains why the settings surface keeps generating silent override bugs. `mode` is not just another control; it is a macro that reapplies a baseline across 13 fields. Everything else is a patch layered on top. The UI does not disclose that hierarchy, so users see a flat control surface while the code enforces an undeclared precedence model.
**Impact:** High architectural risk. This is the common root behind the threshold fight, partial preset recall, stale mode chips, and the coming room-offset migration. As long as semantic actions and partial patches both write into the same flat settings object, every new control will inherit the same hidden dominance rules.
**Proposal:** Stop treating every settings mutation as `Partial<DetectorSettings>`. Add semantic actions or explicit layers: `applyModeBaseline`, `applyRoomAdjustment`, `applyPresetOverlay`, and `applyManualOverride`. Even if the stored state remains flat, the write paths need provenance and ownership so the UI can explain which layer currently owns a value.
**Devil's Advocate:** A flat shallow-merge model is simple and cheap. It only works when each control owns orthogonal fields. DoneWell no longer meets that condition, so the simplicity is now creating state ambiguity instead of reducing it.

## [PHASE 3] Finding: Switching a named room to `custom` preserves inherited absolute thresholds, so Option B needs explicit offset state
**Author:** Codex | **Status:** Open
**Evidence:** Named room preset clicks write absolute thresholds at `components/analyzer/settings/RoomTab.tsx:285-289`. Editing room dimensions later flips `roomPreset` to `custom` but only patches the edited dimension at `components/analyzer/settings/RoomTab.tsx:348-355`. Treatment changes do the same at `components/analyzer/settings/RoomTab.tsx:376-379`, and auto-detected dimensions do it at `components/analyzer/settings/RoomTab.tsx:430-436`. The `custom` preset in `lib/dsp/constants.ts:757-764` defines `feedbackThresholdDb: 30` and `ringThresholdDb: 4`, but that baseline is never applied on these transitions. `types/advisory.ts:333-345` defines no room-offset field, and repo search on 2026-03-25 found no `roomOffset` or `thresholdOffset` setting anywhere else.
**Finding:** A user can select `Arena / Hall`, inherit `38/6` thresholds, tweak one room dimension, and now see `Custom` highlighted while those arena thresholds remain live. That is already a sticky-state bug under the current absolute-preset model. After Don's Option B decision, it becomes a migration blocker because the data model still has no place to store "room as signed offset" separately from the current absolute thresholds.
**Impact:** Medium-to-high implementation risk. The current custom-room path can already misrepresent which sensitivity is active, and a naive conversion to offsets will either lose the inherited behavior silently or preserve it without any explicit state describing where it came from.
**Proposal:** Add explicit room-adjustment state before implementing Option B. Then migrate room presets from absolute thresholds to signed offsets and decide what `custom` means: zero offset by default, or inherited offset with visible provenance. If inherited offset is kept, surface it in the UI instead of hiding it behind the `custom` chip.
**Devil's Advocate:** Preserving the previous room's sensitivity while the engineer fine-tunes dimensions may be useful during setup because it avoids abrupt detector changes. If that is the intended workflow, it still must be explicit. A `custom` chip with invisible inherited thresholds is not an acceptable contract.
