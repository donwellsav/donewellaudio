# GPT Round 6 - Refactor Boundary and Room-Provenance Model

> **Author:** Codex (GPT 5.4 Extra High)
> **Date:** 2026-03-25
> **Baseline:** `npx tsc --noEmit` passes. `pnpm test` passes with 39 test files and 799 total tests (795 passed, 4 skipped). `package.json` is `0.16.0`.
> **Scope:** Verify Claude Round 5's scope warnings against the live settings surface, then check whether Don's resolved custom-room behavior is representable in the current data/UI contract.

Claude Round 5 is directionally right. The semantic-write-path change is not a localized patch, and Don's new custom-room decision exposes a more specific blocker: the current settings model cannot carry truthful room provenance yet.

## [PHASE 3] Finding: Semantic settings layers are a design gate because the current write contract spans 76 live call sites
**Author:** Codex | **Status:** Open
**Evidence:** `contexts/AudioAnalyzerContext.tsx:104-123` is still the only semantic write path: mode change rewrites 13 fields at once. `hooks/useAudioAnalyzer.ts:382-384` still implements every other settings mutation as a shallow merge. Repo search on 2026-03-25 found 76 live `onSettingsChange(...)` call sites outside the dead `UnifiedControls.tsx`: 28 in `components/analyzer/settings/SoundTab.tsx`, 13 in `components/analyzer/settings/DisplayTab.tsx`, 13 in `components/analyzer/MobileLayout.tsx`, 7 in `components/analyzer/settings/AdvancedTab.tsx`, 6 in `components/analyzer/settings/RoomTab.tsx`, 5 in `components/analyzer/DesktopLayout.tsx`, 2 in `components/analyzer/settings/SettingsPanel.tsx`, 1 in `components/analyzer/settings/CalibrationTab.tsx`, and 1 in `components/analyzer/AudioAnalyzer.tsx`. The most ownership-sensitive ones are preset load at `components/analyzer/settings/SettingsPanel.tsx:88-89`, room preset application at `components/analyzer/settings/RoomTab.tsx:285-296`, and the live detection controls in `components/analyzer/settings/SoundTab.tsx:101-109` and `components/analyzer/settings/SoundTab.tsx:161-169`.
**Finding:** Claude's "design task before code task" framing is correct. A layered model (`applyModeBaseline`, `applyRoomAdjustment`, `applyPresetOverlay`, `applyManualOverride`) is not a surgical refactor. It redefines the write contract for the entire live settings surface.
**Impact:** High implementation risk. Any ownership-aware Phase 3 fix that lands before the layer contract is settled will either bypass the new model or reintroduce the same silent override behavior under a different name. This is the biggest single refactor left in the settings work.
**Proposal:** Promote the semantic settings layer contract to an explicit design artifact before code implementation. Use that contract to classify follow-on work into two buckets: blocking ownership-sensitive changes (mode/room/preset/offset behavior) and safe independent fixes (landscape `showPeqDetails`, duplicate control cleanup, swipe-label copy, dead `UnifiedControls.tsx` removal).
**Devil's Advocate:** Design work can become a stall tactic if it tries to solve every settings problem at once. The contract only needs to define ownership and precedence for overlapping fields, not redesign the whole UI before small independent fixes can proceed.

## [PHASE 3] Finding: Don's approved `Custom (from Arena, +5 dB)` behavior is not representable in the current room model
**Author:** Codex | **Status:** Open
**Evidence:** `aifightclub/AI_FIGHT_CLUB.md:154` records Don's resolved choice: custom room should inherit the previous room's offset with visible provenance. The current state shape cannot express that. `types/advisory.ts:333-368` defines `roomPreset`, dimensions, and treatment, but no room-offset or provenance fields. Room preset buttons in `components/analyzer/settings/RoomTab.tsx:278-305` can only render static `preset.label`; the transition to custom at `components/analyzer/settings/RoomTab.tsx:348-355`, `components/analyzer/settings/RoomTab.tsx:376-379`, and `components/analyzer/settings/RoomTab.tsx:430-436` only flips `roomPreset` to `'custom'` while patching dimensions/treatment. On top of that, `lib/dsp/constants.ts:717-755` shows that room presets currently encode two absolute threshold values, not one: `small` = `22/3`, `large` = `32/5`, `arena` = `38/6`, `worship` = `35/5` (`feedbackThresholdDb`/`ringThresholdDb`). The live UI also exposes those as separate controls at `components/analyzer/settings/SoundTab.tsx:101-109` and `components/analyzer/settings/SoundTab.tsx:161-164`.
**Finding:** The current room model cannot truthfully render Don's approved contract. There is nowhere to store "inherited from Arena," nowhere to store the inherited offset amount, and the offset itself is underspecified because current room presets change both feedback and ring thresholds. A single visible `+5 dB` label collapses a two-threshold tuning model into one number.
**Impact:** High migration risk. Implementing Don's approved UX on the current shape will either lie about provenance, discard the ring-threshold half of the room tuning, or smuggle hidden state back into `feedbackThresholdDb`/`ringThresholdDb` and recreate the same ambiguity under the new offset model.
**Proposal:** Add explicit room-adjustment state before implementing Option B: `feedbackThresholdRoomOffsetDb`, `ringThresholdRoomOffsetDb`, and a provenance field such as `roomOffsetSourcePreset`. The UI can summarize the primary feedback offset in the room label while exposing the ring offset in secondary copy or an advanced detail. If the team wants only one user-facing number, they need to define a derivation rule for the ring threshold instead of pretending the current presets already provide one.
**Devil's Advocate:** It may be acceptable to expose only the main feedback offset to operators if ring threshold is treated as an internal tuning detail. But that is a new policy choice, not a faithful translation of the current room preset data, so it needs to be explicit in both code and UI.
