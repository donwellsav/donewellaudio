# GPT Round 3 - Phase 3/4 Control Surface Audit

> **Author:** Codex (GPT 5.4 Extra High)
> **Date:** 2026-03-25
> **Baseline:** `npx tsc --noEmit` passes. `pnpm test` passes with 39 test files and 799 total tests (795 passed, 4 skipped). `package.json` is `0.16.0`.
> **Scope:** Accept Claude Round 2's resolved Phase 1 positions, incorporate Don's answers, and move the audit into Phase 3 (controls/settings) plus one Phase 4 mobile UX defect that surfaced while tracing the live settings path.

Claude Round 2 resolves the previous disputes cleanly enough that I am not spending Round 3 on re-arguing them. The real risk has moved to the operator surface: which control owns the detector state, what a "preset" actually means, and whether the mobile UI honors the same settings the desktop UI exposes.

## [PHASE 3] Finding: Mode and room presets silently fight over the same thresholds
**Author:** Codex | **Status:** Open
**Evidence:** `contexts/AudioAnalyzerContext.tsx:104-118` makes every mode chip rewrite `feedbackThresholdDb`, `ringThresholdDb`, `growthRateThreshold`, `fftSize`, frequency range, sustain, clear, confidence, and prominence; `components/analyzer/settings/RoomTab.tsx:285-296` makes every room preset rewrite `feedbackThresholdDb` and `ringThresholdDb` again while also changing room dimensions and treatment.
**Finding:** The UI presents `Mode` and `Room Preset` as separate concepts, but both mutate the same core sensitivity knobs. A user can tune the detector via mode, then unknowingly overwrite those values from the Room tab, or vice versa, without any indication of which control currently "owns" the thresholds.
**Impact:** Control non-orthogonality. A FOH engineer cannot build a reliable mental model of the detector because venue metadata and program material presets are both writing the same live thresholds. That is exactly how a show starts with "Speech" expectations and ends with ballroom or worship sensitivity because the last tab touched won.
**Proposal:** Separate program profile from venue bias. `Mode` should own detector defaults; `Room` should either (a) feed an explicit derived offset layer, or (b) stop mutating thresholds directly and confine itself to room-physics features. If direct rewrites are kept, the UI needs visible provenance such as "threshold currently overridden by Room Preset: Large Venue."
**Devil's Advocate:** For less technical users, coupled presets are simpler because one room button can make the detector more conservative immediately. The problem is not the existence of the shortcut; it is the absence of any disclosure that two different controls are rewriting the same live state.

## [PHASE 3] Finding: "Save as Preset" does not save the detector state it materially depends on
**Author:** Codex | **Status:** Open
**Evidence:** `components/analyzer/settings/SettingsPanel.tsx:35-43` defines `PRESET_KEYS` as only 13 fields; room settings, `fftSize`, `thresholdMode`, `peakMergeCents`, `harmonicToleranceCents`, `maxTracks`, `trackTimeoutMs`, and all display/canvas settings are excluded. `components/analyzer/settings/SettingsPanel.tsx:69-75` saves only those keys, and `components/analyzer/settings/SettingsPanel.tsx:88-89` reloads the partial object verbatim.
**Finding:** The product calls these saved objects "presets," but they are only a partial detector snapshot. The current room model, FFT size, threshold mode, track limits, harmonic tolerance, and other high-impact advanced state persist outside the saved preset, so recalling a preset does not recreate the system state the operator thought they captured.
**Impact:** False recall confidence. A saved preset can appear to represent a venue or workflow while inheriting incompatible room and DSP settings from whatever was already active. This is especially dangerous because Room Presets themselves rewrite threshold values, yet room state is not part of the saved preset contract.
**Proposal:** Rename the feature to something honest like `Detection Snapshot`, or expand it into a full state preset with explicit sections and opt-in checkboxes (`Detection`, `Venue`, `Display`, `Advanced`). At minimum, the save/load UI must disclose exactly what is and is not captured.
**Devil's Advocate:** Partial presets can be useful when the intent is "keep my preferred detection taste, not the whole rig." If that is the intended product contract, the current labeling and save UX still need to say so explicitly.

## [PHASE 3] Finding: The live settings IA collapses setup, venue, and expert controls into one catch-all tab while duplicating controls across tabs
**Author:** Codex | **Status:** Open
**Evidence:** `components/analyzer/settings/SettingsPanel.tsx:45-48` exposes only two top-level tabs: `Sound` and `Display`. The live `Sound` tab then absorbs Room (`components/analyzer/settings/SoundTab.tsx:232-235`), Calibration (`components/analyzer/settings/SoundTab.tsx:237-241`), Advanced (`components/analyzer/settings/SoundTab.tsx:244-248`), Data Collection (`components/analyzer/settings/SoundTab.tsx:353-357`), and Custom Presets (`components/analyzer/settings/SoundTab.tsx:360-363`). Meanwhile `showThresholdLine` and `faderMode` already exist in `Sound` (`components/analyzer/settings/SoundTab.tsx:111-129`) and are duplicated again in `Display` (`components/analyzer/settings/DisplayTab.tsx:83-99`).
**Finding:** The current live UI is not organized around operator tasks. It is a two-tab shell with a giant `Sound` catch-all that mixes "touch during the show" controls, "set once for this venue" controls, lab/debug tuning, and even privacy consent. On top of that, some controls appear in both tabs, so the user has no clean memory of where the canonical version lives.
**Impact:** High operator friction at FOH. Time-critical controls are buried in a long scroll, setup-only controls are mixed with live-trim controls, and duplicated controls erode trust because the same state is editable from multiple places. This is the opposite of what a tablet-side live tool should optimize for.
**Proposal:** Split the live IA by task, not by implementation history: `Detect` (sensitivity, mode, range, confidence), `Venue` (room, calibration, preset recall), `Display` (graph and card presentation), and `Advanced` (threshold mode, FFT, track limits, debug, data collection). Remove duplicate controls so each concept has one home.
**Devil's Advocate:** A two-tab shell is compact and may feel simpler on paper. In practice, it only works if the `Sound` tab stays narrowly scoped. Once Room, Calibration, Advanced, and preset management all live there, the simplicity is fake.

## [PHASE 3] Finding: Don-approved ML disable cannot be expressed without abandoning Auto mode
**Author:** Codex | **Status:** Open
**Evidence:** Don approved an ML toggle in `aifightclub/CLAUDE_ROUND_2.md:20-21`. The current live algorithm control in `components/analyzer/settings/SoundTab.tsx:253-280` disables the per-algorithm grid whenever `algorithmMode === 'auto'` by adding `pointer-events-none`, and each button handler immediately returns on `isAuto`.
**Finding:** The UI already has an ML chip, but it only works after the user exits Auto mode and takes over all seven algorithms manually. That is not the same product as "disable ML integration." It forces an operator who only wants `ML off` to opt out of the rest of the automatic fusion policy too.
**Impact:** Don's approved control request is blocked by the current interaction model. The app cannot offer a simple "ML on/off" promise while Auto mode monopolizes algorithm selection.
**Proposal:** Add a separate `Use ML filter` toggle that acts as an upper-bound gate across both Auto and Custom modes. Auto should continue to manage the remaining six algorithms when ML is disabled, instead of forcing the operator into full manual algorithm management.
**Devil's Advocate:** Every new top-level toggle increases surface area. Here the toggle is justified because Don explicitly asked for it, and the current workaround is more complex than the control itself.

## [PHASE 4] Finding: Mobile ignores the swipe-labeling toggle and forces gesture mode on
**Author:** Codex | **Status:** Open
**Evidence:** `components/analyzer/settings/DisplayTab.tsx:47-51` exposes `Swipe to Label` as a normal setting. Desktop honors that setting at `components/analyzer/DesktopLayout.tsx:161-162` and `components/analyzer/DesktopLayout.tsx:244-245`. Mobile does not: `components/analyzer/MobileLayout.tsx:257` and `components/analyzer/MobileLayout.tsx:388` pass bare `swipeLabeling`, which is always `true` in JSX. `components/analyzer/IssuesList.tsx:732-782` only renders the mobile button toolbar when `touchFriendly && !swipeLabeling`, so forced-true mobile never shows the non-swipe action buttons.
**Finding:** The setting is not authoritative on mobile. A user can turn `Swipe to Label` off in Display settings and still get the swipe workflow on mobile because the layout hard-wires the prop to true.
**Impact:** Cross-device trust break. The same settings UI claims one behavior, desktop follows it, and mobile silently overrides it. On touch devices that means the user can lose the explicit FALSE+/CONFIRM/X toolbar even after disabling swipe gestures.
**Proposal:** Pass `settings.swipeLabeling` through MobileLayout the same way DesktopLayout already does. If the product truly wants forced gestures on mobile, remove the toggle there or relabel it as desktop-only.
**Devil's Advocate:** The team may have intentionally forced gestures on small screens to save space. If so, the current bug is not the behavior itself - it is exposing a global toggle that lies about what mobile will actually do.
